Lab 1: Booting a PC

Part 1: PC Bootstrap

  • Background:
    • Inline assembly for x86 in Linux, includes:
    • register naming (%eax), source and destination ordering, immediate operand ($0xffff), indirect memory reference
    • Layout of the PC's physical address space, includes where the BIOS is loaded into and where the memory mapped devices (MMIO) lies.
  • Objective:
    • Investigate how an IA-32 compatible computer boots.
  • Methods:
    • Use the QEMU emulator simulating the x86 hardware to boot the compiled kernel.
    • Use the GNU debugger (GDB) to step through the early boot process.
  • Results:
    • Simplified flow is: BIOS (at physical address 0x000ffff0) => bootloader loaded at IP=0x7c00 => bootloader reads the remaining sectors of the kernel from the disk.

Part 2: The Boot Loader

  • The bootloader is composed of boot.S and main.c in source code of lab1.
    • The work of boot.S:
      • cli, cld : disable interrupts
      • Enable A20 gate to enable decoding of memory >= 1MB, and load gtdr to fill the segment descriptors table.
      • Switch from real mode to protected mode, by setting bit 0 of CR0, and ljmp to load new %cs.
      • (in protected mode) Set %esp to setup kernel stack growing down from 0x7c00, and call bootmain (the part of bootloader in C).
    • The work of main.c:
      • Read 4096bytes from the 1st sector of harddisk by accessing the I/O port of ATA bus, and decode the ELF header with loading information.
      • The link address of the kernel is at 0x100000 that is defined in kernel.ld
      • Load the kernel to load address (LMA) defined in ELF segment information.

Part 3: Kernel

  • Map the first 4MB of physical memory using the statically-initialized page directory and page table in kern/entrypgdir.c.
  • Program the code to support formatted printing to the console (refer to lib/printfmt.c).
  • According to the C calling conventions on the x86 and the content of stack when the nested call instructions execute, implement a stack backtrace function (refer to kern/monitor.c).
  • Modify the stack backtrace function to display, for each eip, the function name, source file name, and line number corresponding to that eip (refer to kern/kdebug.c).

Lab 2: Memory management

Part 1: Physical Page Management

  • Implement the physical page allocator (refer to kern/pmap.c), which requires the knowledge of hardware structures to manage memory in x86 hardware.
    • The translation between logical address, linear address, and physical address.
    • Segmentation: global descriptor table (GDT) and local descriptor table (LDT), segment registers, segment selectors, and the fields affecting how the memory controller translates the addresses
    • Virtual address and page table

Part 2: Virtual Memory

  • Write a set of routines to manage page tables: to insert and remove linear-to-physical mappings, and to create page table pages when needed (refer to kern/pmap.c).

Part 3: Kernel Address Space

  • Initialize the kernel address space by completing function mem_init() in kern/pmap.c
    • Set up the address space above UTOP in inc/memlayout.h

Lab 3: User Environments

  • Goal: Initially create one user environment, and then design the JOS kernel to support multiple environments.

Part A: User Environments and Exception Handling

  • a JOS "environment" couples the concepts of "thread" and "address space":
    • The thread is defined by the saved registers (e.g., struct env_tf stands for the trap frame)
    • The address space is defined by the page directory and page tables (i.e., struct env_pgdir)
    • There can be only a JOS environment active in the kernel at a time, so JOS needs only a single kernel stack.
  • Modify mem_init() in kern/pmap.c to allocate and map the struct envs array.
  • Create and run environments:
    • Because we do not yet have a filesystem, we set up the kernel to load a static binary image that is built-in the kernel itself.
      • Refer to the -b binary option on the linker command line in kern/Makefrag
    • Implement the functions in env.c to set up a JOS user environment:
      • Configures the segmentation hardware with separate segments for privilege level 0 (kernel) and privilege level 3 (user). (refer to env_init() in env.c)
        • With the knowledge of 2-level virtual address translation: PDE (entry of page directory) and PTE (entry of page table)
      • Allocate a page directory for a new environment and initialize the kernel portion of the new environment's address space. (refer to env_setup_vm() in env.c)
      • Parse an ELF binary image, and load its contents into the user address space of a new environment. (refer to load_icode() in env.c)
  • Handle interrupts and exceptions:
    • Implement basic exception and system call handling for kernel to recover control of the processor from user-mode code.
      • With the knowledge of the x86 interrupt and exception mechanism
    • An interrupt is a protected control transfer that is caused by an asynchronous event usually external to the processor.
      • The interrupt descriptor table (IDT):
        • Interrupt vectors 0-255, the vector is used as an index into the processor's interrupt descriptor table
        • an IDT entry contains CS, EIP of the instruction of interrupt handler; bit 0-1 of CS stands for the privilege level in which the instruction executes.
      • The task state segment (TSS):
        • A structure specifies the segment selector and address where the stack lives.
        • JOS only uses TSS to define the kernel stack that the processor should switch to when it transfers from user to kernel mode. The processor uses the ESP0 and SS0 fields of the TSS to define the kernel stack when entering kernel mode.
    • Set up the IDT:
      • Edit trapentry.S and trap.c to initialize the IDT with the addresses of the interrupt handlers, and add an entry point in trapentry.S for each trap defined in inc/trap.h.
      • Test the trap handling code using some of the test programs in the user directory that cause exceptions, such as user/divzero.

Part B: Page Faults, Breakpoints Exceptions, and System Calls

  • Handle page faults:
    • Modify trap_dispatch() in kern/trap.c to dispatch page fault exceptions to page_fault_handler().
      • The page fault exception is handled in interrupt vector 14 (T_PGFLT) that stores the linear address that caused the fault in a special processor control register, CR2.
  • The breakpoint exception:
    • Modify trap_dispatch() in kern/trap.c to make breakpoint exceptions invoke the kernel monitor.
      • The breakpoint exception is handled in interrupt vector 3 (T_BRKPT). Debuggers insert breakpoints in a program's code by temporarily replacing the relevant program instruction with the special 1-byte int3 software interrupt instruction.
  • System calls:
    • Edit kern/trapentry.S and trap_init() in kern/trap.c to add a handler in the kernel for interrupt vector T_SYSCALL, and change trap_dispatch() to handle the system call interrupt by calling syscall().
      • Use int $0x30 as the system call interrupt, and set up the interrupt descriptor to allow user processes to cause the interrupt.
      • The application passes the system call number and the system call arguments in registers. The kernel passes the return value back in %eax.
  • User-mode startup:
    • Modify libmain() in lib/libmain.c to initialize the global pointer thisenv to point at the environment's struct Env in the envs[] array.
      • A user program starts running at the top of lib/entry.S.
    • See that user/hello print "hello, world" and then print "i am environment 00001000", then attempts to exit by calling sys_env_destroy().
      • The kernel should report that it has destroyed the only environment and then drop into the kernel monitor.
  • Page faults and memory protection:
    • Operating systems usually rely on hardware support to implement memory protection.
    • The OS keeps the hardware informed about which virtual addresses are valid and which are not.
    • When a program tries to access an invalid address or one for which it has no permissions, the processor stops the program at the instruction causing the fault and then traps into the kernel with information about the attempted operation.
    • As an example of a fixable fault, consider an automatically extended stack:
      • The kernel initially allocates a single stack page, and then if a program faults accessing pages further down the stack, the kernel will allocate those pages automatically and let the program continue.
    • Change kern/trap.c to panic if a page fault happens in kernel mode.
      • Check the low bits of the struct tf_cs
    • Implement user_mem_check() in kern/pmap.c
      • Read user_mem_assert in kern/pmap.c
    • Change kern/syscall.c to check arguments to system calls
      • Boot the kernel, running user/buggyhello. The environment should be destroyed, and the kernel should not panic.
    • Change debuginfo_eip in kern/kdebug.c to call user_mem_check on usd, stabs, and stabstr.
      • Run user/breakpoint, and run backtrace from the kernel monitor and see the backtrace traverse into lib/libmain.c.
    • Boot the kernel, running user/evilhello. The environment should be destroyed, and the kernel should not panic.
